## 一、先把实验口径彻底定死

这次主实验统一采用你已经确认的方案：

- OpenClaw 固定为 **2026.4.14**
  
- OpenViking runtime 固定为 **0.3.8**
  
- OpenViking 插件走 **`contextEngine=openviking`**
  
- LoCoMo 数据集固定为你清洗后的 **1540 QA 子集**
  
- `openclaw-eval` 继续复用，但先做一轮小修补
  
- 四组实验共用同一前台评测模型、同一 judge、同一数据集，只改 **slot / backend**，不再混入旧版 `slot.memory` 的 `0.1.18` 方案。([GitHub](https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/VERSIONS.lock "openclaw-openviking-doubao/VERSIONS.lock at main · DANNHIROAKI/openclaw-openviking-doubao · GitHub"))
  

四组实验建议固定为：

|组别|目标|contextEngine|memory|
|---|---|---|---|
|G1|OpenClaw(memory-core)|legacy|memory-core|
|G2|OpenClaw + LanceDB (-memory-core)|legacy|memory-lancedb|
|G3|OpenClaw + OpenViking Plugin (-memory-core)|openviking|none|
|G4|OpenClaw + OpenViking Plugin (+memory-core)|openviking|memory-core|

这样做和 OpenClaw 当前架构是对齐的：`contextEngine` 决定模型最终看到什么上下文，`memory` 插件负责 search/retrieval，这两条机制是分开的，也可以协同工作。OpenClaw 文档明确写了，`legacy` 是默认 context engine，`plugins.slots.memory` 和 `plugins.slots.contextEngine` 是两套独立 slot。([OpenClaw](https://docs.openclaw.ai/concepts/context-engine "https://docs.openclaw.ai/concepts/context-engine"))

有一个很重要的取舍：**G1/G2 不要只是把 `contextEngine` 切回 `legacy`，还保留 OpenViking 插件处于启用状态。**因为 OpenViking 当前插件不是“只有 context-engine 的单一插件”，它还同时扮演 hook layer、tool provider 和 local runtime manager。为了让 G1/G2 成为真正干净的 baseline，最稳妥的做法是 **G1/G2 根本不要加载或启用 OpenViking 插件**；只让 G3/G4 装并启用它。这个结论是基于当前插件 README 对插件职责的公开描述作出的实验设计选择。([GitHub](https://github.com/volcengine/OpenViking/blob/main/examples/openclaw-plugin/README.md "https://github.com/volcengine/OpenViking/blob/main/examples/openclaw-plugin/README.md"))

## 二、你说的“一组一个 workdir”，真正应该怎么落地

这里要先讲清一个概念：对 OpenClaw 来说，真正需要隔离的不是“某个随便建的文件夹”，而是 **profile / state dir / workspace / gateway 端口**。官方多 Gateway 文档直接建议：同一台机器上要分离多个实例时，要把 **profile / config、state dir、workspace、base port** 都隔离开；而 agent workspace 文档也明确说，workspace 只是 agent 的工作目录，真正的 config、credentials、sessions 都在 `~/.openclaw` 这一类 state dir 下面。([OpenClaw](https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways "https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways"))

所以，我建议你把“每组一个 workdir”落成下面这种结构：

```text
~/ov-bench/
  g1/
    state/
    workspace/
    outputs/
    logs/
  g2/
    state/
    workspace/
    outputs/
    logs/
  g3/
    state/
    workspace/
    openviking/
    outputs/
    logs/
  g4/
    state/
    workspace/
    openviking/
    outputs/
    logs/
```

实际运行时，不要只换目录名，而要明确给每组配置：

- 一个独立的 OpenClaw state dir
  
- 一个独立的 workspace
  
- 一个独立的 profile 名
  
- 一个独立的 gateway 端口
  
- G3/G4 再各自配一个独立的 `OPENVIKING_HOME`
  

如果你真的想把这件事做“傻瓜化”，最简单的方式是每组一个 profile，例如 `g1/g2/g3/g4`。OpenClaw 文档说明了 profile 会自动 scope 到不同的 state dir，默认会形成类似 `~/.openclaw-<profile>` 的隔离效果。([OpenClaw](https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways "https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways"))

我还要提醒你一个脚本层面的坑：你当前 `bootstrap.sh` 虽然支持 `OPENCLAW_PREFIX` 和 `OPENVIKING_HOME`，但它仍然把 `.env` 和 `openviking.env` 写到 **`$HOME/.openclaw`**，而不是 profile/state-dir 作用域内；同时它还会无条件安装 OpenViking 并把 `plugins.slots.contextEngine` 切到 `openviking`。所以如果你不改它，所谓“一组一个 workdir”其实是假的，组间仍会串配置。([GitHub](https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh "https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh"))

## 三、你现有的 `openclaw-openviking-doubao` 仓库怎么改，才适合这次四组实验

这次我不建议你重写一套新脚本，而是建议你**只对现有 bootstrap 做一次小改造**，然后四组都用它。

你现在这份脚本已经做了最难的事：它会安装 OpenClaw 2026.4.14、做非交互 onboarding、给 OpenViking 建 venv、安装 `openviking==0.3.8`、生成 `ov.conf`、固定 OpenViking 的 embedding 和 VLM、安装 vendored 的 OpenViking plugin snapshot，并把 `contextEngine` 切到 `openviking`。这说明它非常适合作为 **G3/G4 的安装底座**。([GitHub](https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh "https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh"))

你只需要加 4 个开关：

第一，给脚本加一个 **实验模式**，例如 `EXPERIMENT_KIND=base` 和 `EXPERIMENT_KIND=openviking`。  
`base` 只跑 OpenClaw 安装和 onboarding；`openviking` 才继续跑 OpenViking runtime / plugin / contextEngine 配置。这样 G1/G2 用 `base`，G3/G4 用 `openviking`。

第二，让脚本 **严格尊重 profile/state-dir**。  
把现在写死到 `$HOME/.openclaw/.env` 和 `$HOME/.openclaw/openviking.env` 的地方，改成写到当前组对应的 state dir 里。否则 profile 隔离只做了一半。这个修改是必要的，因为 OpenClaw 的 state dir 才保存 config、auth、sessions。([GitHub](https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh "https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh"))

第三，给脚本加 **端口参数**。  
至少要支持 `OPENCLAW_GATEWAY_PORT`；如果你想并行跑多个组，再加 `OPENVIKING_PORT`。OpenClaw 的多 Gateway 文档建议不同实例的 base port 至少错开 20，避免派生端口冲突。即便你最终按顺序跑，我也建议把四组端口固定成不同值，减少服务残留带来的混淆。([OpenClaw](https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways "https://docs.openclaw.ai/ja-JP/gateway/multiple-gateways"))

第四，给脚本加 **workspace 参数**。  
因为 OpenClaw 文档明确区分了 workspace 与 state dir；你想做到“一组一个 workdir”，workspace 也必须是每组独立目录。([OpenClaw](https://docs.openclaw.ai/concepts/agent-workspace "https://docs.openclaw.ai/concepts/agent-workspace"))

如果你按这个思路改，脚本职责就很清楚：

- G1/G2：只负责装 OpenClaw、配置网关认证、设置前台模型、准备独立 state/workspace
  
- G3/G4：在上面基础上再加 OpenViking runtime、`ov.conf` 和 OpenViking plugin

## 六、`openclaw-eval` 这次一定要修的 5 个点

这部分我建议你 fork 一份自己的实验分支，别直接在 upstream 上跑。

### 1）修 user 绑定 bug

当前 `run_ingest()` 默认 user 是 `"eval-1"`，`run_sample_qa()` 默认 user 却是 `f"eval-{sample_idx}"`。README 又要求 QA 必须使用 prior ingest 的同一个 user UUID。这个默认值组合会让评测语义直接崩掉。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/README.md "openclaw-eval/README.md at main · ZaynJarvis/openclaw-eval · GitHub"))

这次最稳的修法不是“提醒自己总是手动传 `--user`”，而是**直接改脚本**：

- ingest 和 qa 都统一按 `group + sample_id` 自动生成 user
  
- 例如 `g3-conv-26`、`g4-conv-50`
  
- 默认 user 不再依赖 `eval-1` 或循环序号
  

这样你就可以整批运行 10 个 sample，而不会串 sample，也不用每次手抄 UUID。

### 2）修输出命名

当前 QA 输出写成 `args.output.{sample_idx}.jsonl`。这会导致结果文件依赖“这次加载列表里的第几个样本”，可读性很差。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))

直接改成按 `sample_id` 命名，例如：

- `qa.conv-26.jsonl`
  
- `qa.conv-30.jsonl`
  

这样后面 merge、judge、查错都容易很多。

### 3）去掉 reset-session hack

当前 `eval.py` 会读 `~/.openclaw/agents/main/sessions/sessions.json`，然后把对应 `.jsonl` 文件改名归档。与此同时，`http-api-setup.md` 已经明确说了：API 场景下最稳的 reset 方式其实是 **换 `user` 或换 session key**；同一个 `user` 才会稳定映射到同一个 session。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))

这次实验里你已经决定 **每个 group × sample 都用唯一 user**，所以这套本地重命名 session 文件的 hack 没有必要，直接删掉或加开关禁用。这样反而更干净。

### 4）不要用 `--viking`

`eval.py` 里的 `--viking` 分支会直接调用 `ov add-memory`，也就是绕开 OpenClaw，把内容直接写入 OpenViking。这条路不能用来评测 “OpenClaw + OpenViking 插件”。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))

这次四组实验全部统一走 OpenClaw 的 `/v1/responses` 路径。

### 5）补一个 merge + summarize 脚本

`judge_util.py` 已经支持直接读取 `.jsonl`；`judge.py` 会按 `correct / total` 计算分数；QA 记录里也已经包含 `usage`。所以你只需要在自己的 fork 里补一个很小的 merge 脚本：

- 把 10 个 sample 的 `qa.<sample_id>.jsonl` 合并成一个 `answers.all.jsonl`
  
- 顺手统计 `usage.input_tokens`
  
- 输出一个 `summary.json`
  

这样 judge 每组只跑一次。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/judge_util.py "openclaw-eval/judge_util.py at main · ZaynJarvis/openclaw-eval · GitHub"))

## 七、LanceDB 专项：这组一定要单独把控

这一组是最容易翻车的，所以我单独给你写。

`openclaw-eval` 自带的 `http-api-setup.md` 其实已经在暗示这组不是“开箱即用”：它写的是先“更新 plugin files”，再把 `memory` slot 切到 `memory-lancedb`，然后配置 embedding、`autoCapture: true`、`autoRecall: true`，最后检查 `~/.openclaw/memory/lancedb/memories.lance/data` 是否存在。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/docs/http-api-setup.md "openclaw-eval/docs/http-api-setup.md at main · ZaynJarvis/openclaw-eval · GitHub"))

更重要的是，OpenClaw 近几周的公开 issue 连续报告过好几类 `memory-lancedb` 打包/依赖问题：

- fresh install / update 后缺少 `@lancedb/lancedb`
  
- `memory-lancedb` 在运行时去找 `dist/package.json` 但找不到
  
- 某些版本需要手动恢复 extension 目录里的依赖后才能加载成功。([GitHub](https://github.com/openclaw/openclaw/issues/44125 "https://github.com/openclaw/openclaw/issues/44125"))
  

所以我对 G2 的建议是：

**不要把 LanceDB 安装混在正式 benchmark 当天临时处理。先把 G2 单独做成一个“安装-验证-冻结”的步骤。**

你具体应该这么做：

1. 先在 G2 的独立 profile/state dir 上装好 **和其他组完全相同版本的 OpenClaw 2026.4.14**。
   
2. 不装 OpenViking。
   
3. 启用 `memory-lancedb`，填上 `doubao-embedding-vision-251215`、你的 API Key、`autoCapture: true`、`autoRecall: true`。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/docs/http-api-setup.md "openclaw-eval/docs/http-api-setup.md at main · ZaynJarvis/openclaw-eval · GitHub"))
   
4. 重启 gateway。
   
5. 立刻检查：
   
    - `openclaw plugins list`
      
    - `openclaw plugins inspect memory-lancedb`
      
    - gateway 日志里是否有 load error
      
    - `~/.openclaw/.../memory/lancedb/memories.lance/data` 是否真的出现。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/docs/http-api-setup.md "openclaw-eval/docs/http-api-setup.md at main · ZaynJarvis/openclaw-eval · GitHub"))
      

如果这一步通过，G2 就冻结，不再升级 OpenClaw，也不再动 node_modules。  
如果没通过，再做修复；但修复也要在 **正式评测之前** 一次性完成，然后重新做 smoke test，再开始烧 API。因为 recent issues 说明 LanceDB 的问题常常和 install/update 过程有关，不是你跑到一半才会出现的逻辑 bug。([GitHub](https://github.com/openclaw/openclaw/issues/44125 "https://github.com/openclaw/openclaw/issues/44125"))

## 八、发烟测试怎么做，才最省钱也最有效

全量 1540 之前，我建议你做三层发烟测试。

### 第一层：会话连续性 smoke

这个和 LoCoMo 无关，只是验证网关 session 机制。

用 `/v1/responses`：

1. `user=smoke-a` 发送“我叫 Dann。”
   
2. 还是 `user=smoke-a` 再问“我叫什么？”
   
3. 换成 `user=smoke-b` 再问“我叫什么？”
   

`http-api-setup.md` 明确写了：**same user = same session**；同时 `/v1/responses` 和 `/v1/chat/completions` 的 session 还不是共享的。`eval.py` 本身也确实是把消息发到 `/v1/responses`，而且 payload 里的 `model` 固定是 `"openclaw"`。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/docs/http-api-setup.md "openclaw-eval/docs/http-api-setup.md at main · ZaynJarvis/openclaw-eval · GitHub"))

这一步只要没通过，后面一律别跑。

### 第二层：每组 1 个 sample 的最小 smoke

四组都跑，但只拿 **1 个 sample**。

建议：

- ingest：只喂前 2 个 session
  
- qa：只问前 5 个题
  
- judge：只打一遍
  

这样你可以很快发现：

- G1 的 baseline 通不通
  
- G2 的 LanceDB 是不是真的在 autoCapture/autoRecall
  
- G3/G4 的 OpenViking plugin 有没有成功接管 `contextEngine`
  
- 你的 user 绑定 patch 有没有生效
  

### 第三层：OpenViking 组的增强 smoke

对 G3/G4 再做一轮加强版：

- 用同一个 sample
  
- ingest 它的全部 session
  
- 再跑前 20 个 QA
  
- 同时盯 OpenClaw 日志和 OpenViking 日志
  

因为 OpenViking 当前插件的关键行为都发生在 `before_prompt_build`、`assemble()`、`afterTurn()` 和 `compact()` 这一整条生命周期里；README 也明确说它会在 local mode 下启动并监控 OpenViking subprocess。([GitHub](https://github.com/volcengine/OpenViking/blob/main/examples/openclaw-plugin/README.md "https://github.com/volcengine/OpenViking/blob/main/examples/openclaw-plugin/README.md"))

## 九、正式全量 1540 的执行方式

我建议你 **按组顺序串行跑**，不要四组并行。  
顺序建议是：**G1 → G2 → G3 → G4**。原因很简单：G2 安装风险最高，G3/G4 API 花费更大，先把 baseline 和 LanceDB 跑通，再上 OpenViking。

每一组的正式运行都按同样流程：

### 第一步：准备干净环境

这一组对应的 profile/state dir/workspace 必须是干净的，不能残留之前的 smoke 数据。OpenClaw 文档也反复强调，迁移/重置时 state dir 里保存的是 config、auth、sessions 和 workspace，所以隔离一定要做彻底。([OpenClaw](https://docs.openclaw.ai/install/migrating "https://docs.openclaw.ai/install/migrating"))

### 第二步：验证当前组的 slot 配置

正式跑前再检查一次：

- G1：`contextEngine=legacy`，`memory=memory-core`
  
- G2：`contextEngine=legacy`，`memory=memory-lancedb`
  
- G3：`contextEngine=openviking`，`memory=none`
  
- G4：`contextEngine=openviking`，`memory=memory-core`
  

同时再看一眼 `openclaw plugins list`。  
G1/G2 里不要有启用中的 OpenViking plugin；G3/G4 里 OpenViking plugin 必须启用且健康。

### 第三步：按 sample 跑 ingest

数据集的 10 个 sample ID，你的数据脚本已经固定成：

`conv-26, conv-30, conv-41, conv-42, conv-43, conv-44, conv-47, conv-48, conv-49, conv-50`。([GitHub](https://github.com/DANNHIROAKI/OpenViking-LoCoMo10/blob/main/Load_Data.sh "https://github.com/DANNHIROAKI/OpenViking-LoCoMo10/blob/main/Load_Data.sh"))

每个 sample 都应该绑定自己的 user，例如：

- G1 用 `g1-conv-26`
  
- G2 用 `g2-conv-26`
  
- G3 用 `g3-conv-26`
  
- G4 用 `g4-conv-26`
  

然后 ingest 这个 sample 的全部 session。  
不要跨 sample 复用 user。

### 第四步：按同一 user 跑该 sample 的 QA

同一个 sample 的 QA 题，必须继续落到 ingest 时的同一个 user 上。README 写得非常明确：ingest 是为了 build up memory/context，qa 是在 prior ingest 的基础上提问。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/README.md "openclaw-eval/README.md at main · ZaynJarvis/openclaw-eval · GitHub"))

### 第五步：一组跑完 10 个 sample 后，再统一 merge 和 judge

这时你会得到这一组的：

- 10 个 sample 级 JSONL
  
- 1 个组级 merged JSONL
  
- 1 个 judge 输出 JSON
  
- 1 个 token 汇总 JSON
  

judge 只跑一次即可。`judge.py` 的计算很直接：`correct = 判为 CORRECT 的题数`，`total = 参与打分的题数`，`score = correct / total`。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/judge.py "openclaw-eval/judge.py at main · ZaynJarvis/openclaw-eval · GitHub"))

## 十、最后结果怎么汇总，才不含糊

`openclaw-eval` 当前脚本里，QA 记录会保存：

- `question`
  
- `expected`
  
- `response`
  
- `category`
  
- `evidence`
  
- `usage`
  

而 judge 会根据 `question + expected + response` 做宽松打分；它明确写了，只要 generated answer 和 gold answer 说的是同一主题，或者时间问题指向同一日期/时间段，就可以算 CORRECT。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))

所以最终至少要出两类指标：

### 1）完成率

直接用 judge 输出：

**Task Completion Rate = correct / total**

全量时，`total` 就是 1540。  
这其实就是 LoCoMo QA 的 judge-based accuracy。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/judge.py "openclaw-eval/judge.py at main · ZaynJarvis/openclaw-eval · GitHub"))

### 2）token 成本

这里我建议你**同时报两列**，因为当前公开脚本对 QA 的 token 汇总更直接，而 ingest 也确实会产生 usage，但没有做同样显式的总表输出。

- **QA Input Tokens Total**：把 QA 阶段所有记录里的 `usage.input_tokens` 求和。`run_qa()` 当前就会把它累计成 `total_usage['input_tokens']` 并写 summary。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))
  
- **Full Pipeline Input Tokens Total**：把 ingest 和 QA 两阶段的 `usage.input_tokens` 都加起来。因为 ingest 里每个 session 请求同样会记录 usage。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))
  

我建议你最后的总表直接做成两版：

第一版，和当前 `openclaw-eval` 更贴近的 **QA-only** 表。  
第二版，实验解释更完整的 **Full-pipeline** 表。

这样以后别人问“你这个 token 成本到底算没算 ingest”，你不会陷入歧义。

## 十一、我建议你最后交付哪些文件

每组都留这些东西：

- `config.snapshot.json`：这一组开跑前的 OpenClaw config 快照
  
- `qa.<sample_id>.jsonl`：10 个 sample 的原始 QA 记录
  
- `answers.all.jsonl`：组级合并文件
  
- `judge.json`：judge 输出
  
- `usage.json`：QA-only 与 full-pipeline 两套 token 汇总
  
- `openclaw.log`：跑全量期间的 gateway 日志
  
- `openviking.log`：G3/G4 的 OpenViking 日志
  
- `README-run.md`：这一组的端口、profile、state dir、workspace、模型、embedding、judge 信息
  

最后再出一个总表，至少包含：

- Group
  
- Completion Rate
  
- QA Input Tokens Total
  
- Full Input Tokens Total
  
- Notes
  

## 十二、最适合你的实际执行顺序

按这个顺序做，成功率最高：

1. 先 fork `openclaw-eval`，做 user 绑定、输出命名、禁用 reset hack、merge/summarize 这四个小修改。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/eval.py "openclaw-eval/eval.py at main · ZaynJarvis/openclaw-eval · GitHub"))
   
2. 把你的 bootstrap 脚本改成支持 `base/openviking` 两种模式，并让它真正尊重 profile/state-dir/workspace。([GitHub](https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh "https://github.com/DANNHIROAKI/openclaw-openviking-doubao/blob/main/scripts/bootstrap.sh"))
   
3. 先单独把 G2 的 LanceDB 安装和验证做完，确认它不再有 load/dependency 问题。([GitHub](https://github.com/ZaynJarvis/openclaw-eval/blob/main/docs/http-api-setup.md "openclaw-eval/docs/http-api-setup.md at main · ZaynJarvis/openclaw-eval · GitHub"))
   
4. 跑三层 smoke test。
   
5. 依次跑 G1、G2、G3、G4 的全量 1540。
   
6. 每组跑完立刻 merge、judge、封存日志。
   
7. 最后再生成总表和实验报告。
